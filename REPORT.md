## Раздел 1: Описание проблемы

### Что такое Race Condition?

Race condition (состояние гонки) — это ситуация, когда несколько транзакций параллельно обрабатывают (меняют) одни и те же данные и конечный результат будет зависеть от того, в каком порядке эти транзакции будут выполнены или в принципе есть риск получить невалидный результат. 

**Пример из жизни:**
- повторная оплата заказа
- покупка последнего билета (кинотеатр, самолет и тд), два пользователя одновременно могли выбрать место и начать процедуру оплаты

### Почему READ COMMITTED не защищает от двойной оплаты?

На уровне изоляции READ COMMITTED:
1. Каждая транзакция видит только результаты полностью завершённых транзакций и не видит результаты транзакций, которые выполняются сейчас параллельно.
2. Когда две транзакции одновременно используют одни и те же данные, они ничего не знают друг о друге
3. В результате если первая транзакция внесёт изменения в данные (например, будет куплен последний билет), то вторая транзакция ничего об этом знать не будет, потому что получила данные до завершения первой. В итоге вторая транзакция тоже может "купить последний билет", в результате чего один билет достанется двум покупателям.

**Демонстрация проблемы:**

```
Время | Сессия 1                    | Сессия 2
------|----------------------------|---------------------------
t1    | BEGIN                      |
t2    | SELECT status (created)    |
t3    |                            | BEGIN
t4    |                            | SELECT status (created)
t5    | pg_sleep(3)                |
t6    |                            | pg_sleep(3)
t7    | UPDATE status = paid       |
t8    |                            | UPDATE status = paid (ждет)
t9    | COMMIT                     |
t10   |                            | UPDATE выполняется...
t11   |                            | COMMIT
```

_TODO: Объясните, что происходит на каждом этапе._
1. Сначала сессия 1 берёт запись, статус которой = 'created' (t1-t2)
2. Пока сессия 1 еще не закрылась открылась вторая сессия, которая обратилась к той же записи, которая всё еще имеет статус 'created' (t3-t4)
3. Обе сессии (транзакции) ничего не знают друг о друге и работают с одной записью, подразумевая, что она имеет статус 'created'
4. Первая сессия временно блокирует доступ к записи и обновляет её статус до 'paid' (t7)
5. Вторая сессия пытается обновить статус до 'paid', но пока запись еще заблокирована (выполняется update из сессии 1) - сессия в состоянии ожидания (t8)
6. Сессия 1 заканчивает свой update снимает блокировку и завершает свою работу (t9)
7. После коммита первой сессии, вторая выполняет свой апдейт, который будет успешно выполнен - > в результате две записи об оплате (t10)

## Раздел 2: Уровни изоляции в PostgreSQL

### READ UNCOMMITTED

**Описание:**
_TODO: Опишите этот уровень изоляции._
Данный уровень полностью исключает изоляцию как таковую - каждая транзакция видит и учитывает все промежуточные результаты всех других транзакций даже до их завершения. 

**Предотвращает:**
- никакие аномалии не предотвращаются

**Не предотвращает:**
- Dirty reads
- Non-repeatable reads
- Phantom reads

**Когда использовать:**
По возможности никогда.

**Особенность в PostgreSQL:**
В PostgreSQL READ UNCOMMITTED работает как READ COMMITTED из-за архитектуры MVCC.

---

### READ COMMITTED (по умолчанию)

**Описание:**
_TODO: Опишите этот уровень изоляции._

Каждый SELECT в транзакции видит результаты только успешно завершённых транзакций.

**Предотвращает:**
- ✅ Dirty reads (чтение незакоммиченных данных)

**Не предотвращает:**
- ❌ Non-repeatable reads (повторное чтение дает другой результат)
- ❌ Phantom reads (новые строки появляются в результате запроса)

**Пример non-repeatable read:**
```sql
-- Сессия 1
BEGIN;
SELECT balance FROM accounts WHERE id = 1; -- Результат: 1000
-- Сессия 2 изменяет balance и делает COMMIT
SELECT balance FROM accounts WHERE id = 1; -- Результат: 500 (!)
COMMIT;
```

**Когда использовать:**
В простых CRUD приложениях.

---

### REPEATABLE READ

**Описание:**
_TODO: Опишите этот уровень изоляции._

Транзакция видит snapshot данных на момент начала выполнения первого запроса, при этом в случае повторного запроса внутри транзакции будет использован тот же snapshot, даже если другая транзакция закоммитила изменения этих данных.

**Предотвращает:**
- ✅ Dirty reads
- ✅ Non-repeatable reads
- ✅ Phantom reads (в PostgreSQL благодаря MVCC)

**Не предотвращает:**
- ❌ Write skew (специфичная аномалия)
- ❌ Некоторые сериализационные аномалии

**Особенность в PostgreSQL:**
В отличие от стандарта SQL, PostgreSQL на REPEATABLE READ также предотвращает phantom reads.

**Когда использовать:**
Сложные расчёты внутри одной транзакции, требующие корректного неизменяемого набора данных. 

---

### SERIALIZABLE

**Описание:**
_TODO: Опишите этот уровень изоляции._

Самый строгий уровень, гарантирующий, что параллельные транзакции будут выполнены так, как если бы они выполнялись последовательно.

**Предотвращает:**
- ✅ Все аномалии чтения
- ✅ Сериализационные аномалии
- ✅ Write skew

**Недостатки:**
- ❌ Может откатывать транзакции при конфликтах (serialization failure)
- ❌ Снижение производительности
- ❌ Требует retry logic в приложении

**Пример serialization failure:**
```sql
-- Сессия 1
BEGIN ISOLATION LEVEL SERIALIZABLE;
SELECT SUM(balance) FROM accounts; -- 5000
UPDATE accounts SET balance = balance + 100 WHERE id = 1;

-- Сессия 2 (параллельно)
BEGIN ISOLATION LEVEL SERIALIZABLE;
SELECT SUM(balance) FROM accounts; -- 5000
UPDATE accounts SET balance = balance + 200 WHERE id = 2;

-- При COMMIT одна из транзакций получит ошибку:
-- ERROR: could not serialize access due to read/write dependencies
```

**Когда использовать:**
В самых требовательных сценариях: денежные переводы, бронирование.

## Раздел 3: Решение проблемы

### Почему REPEATABLE READ решает проблему?

_TODO: Объясните, как REPEATABLE READ помогает решить проблему конкурентных оплат._

REPEATABLE READ использует snapshot isolation:
1. Транзакция видит состояние одного и того же snapshot во всех своих select запросах.
2. Это означает, что если параллельно вторая транзакция внесет изменения в запись, с которой работает первая, то первая это изменение не увидит даже при повторном select запросе.
3. Однако, без блокировок обе транзакции будут параллельо работать с записью, думая что она имеет статус 'created' и обе попытаются выполнить update
4. Тем не менее после заверешния первой транзакции, вторая не сможет выполнить операцию update на той же записи, поскольку СУБД выявит что эта запись уже была изменена.

### Зачем нужен FOR UPDATE?

_TODO: Объясните роль FOR UPDATE в решении проблемы._
`FOR UPDATE` блокирует строку которая нужна для транзакции, что гарантирует, что другие транзакции не смогут внести изменения в эту строку до заверешения первой

`FOR UPDATE` создает эксклюзивную блокировку на уровне строки (row-level lock):

**Без FOR UPDATE:**
```sql
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT status FROM orders WHERE id = '...';
-- Другая транзакция может прочитать ту же строку
UPDATE orders SET status = 'paid' WHERE id = '...';
COMMIT;
```

**С FOR UPDATE:**
```sql
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT status FROM orders WHERE id = '...' FOR UPDATE;
-- Другая транзакция ЖДЕТ освобождения блокировки
UPDATE orders SET status = 'paid' WHERE id = '...';
COMMIT;
```

**Типы блокировок:**
- `FOR UPDATE` — эксклюзивная блокировка (никто не может изменить или заблокировать)
- `FOR SHARE` — разделяемая блокировка (другие могут читать с FOR SHARE, но не могут изменять)
- `FOR NO KEY UPDATE` — как FOR UPDATE, но разрешает concurrent FOR KEY SHARE
- `FOR KEY SHARE` — слабая блокировка (предотвращает DELETE и UPDATE ключевых полей)

### Что произойдет без FOR UPDATE на REPEATABLE READ?

Даже на REPEATABLE READ возможна аномалия:

```sql
-- Сессия 1
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT status FROM orders WHERE id = '...'; -- created
-- Задержка...
UPDATE orders SET status = 'paid' WHERE id = '...' AND status = 'created';
COMMIT;

-- Сессия 2 (параллельно)
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT status FROM orders WHERE id = '...'; -- created (snapshot!)
-- Задержка...
UPDATE orders SET status = 'paid' WHERE id = '...' AND status = 'created';
-- ЧТО ПРОИЗОЙДЕТ?
```

Без FOR UPDATE заказ будет оплачен только один раз (сессия 1), но сессия 2 попытается обновить статус и вызовет ошибку на этапе update. То есть сессия 2 будет использовать неактуальные данные и завершится ошибкой, которую нужно обработать.

### Разница между FOR UPDATE и FOR SHARE

| Характеристика | FOR UPDATE | FOR SHARE |
|----------------|------------|-----------|
| Тип блокировки | Эксклюзивная | Разделяемая |
| Блокирует чтение с FOR UPDATE | ✅ | ✅ |
| Блокирует чтение с FOR SHARE | ✅ | ❌ |
| Блокирует UPDATE | ✅ | ✅ |
| Блокирует DELETE | ✅ | ✅ |
| Use case | Перед изменением | Защита от изменений |

**Пример использования FOR SHARE:**
```sql
-- Проверка наличия товара перед созданием заказа
SELECT quantity FROM products WHERE id = '...' FOR SHARE;
-- Товар не может быть удален или изменен до COMMIT
```

## Раздел 4: Рекомендации для продакшена

### Какой ISOLATION LEVEL использовать для продакшена маркетплейса?

**Рекомендация:** Для продакшена маркетплейса рекомендуется использовать **READ COMMITTED как default уровень** с явным использованием **REPEATABLE READ + FOR UPDATE для критичных операций**.

#### Обоснование:

**1. Производительность**
- READ COMMITTED имеет минимальный overhead, поскольку не выполняет проверку на изменения в записях, фантомные записи и тд, что снижает нагрузку
- REPEATABLE READ используется только там, где необходимо, что уменьшает затраты на анализ изменений в записях и обработку ошибок при параллельных апдейтах
- Избегаем глобального использования SERIALIZABLE, поскольку это будет увеличивать время простоев и требовать обработку ошибок

**2. Безопасность данных**
- Критичные операции (оплата, изменение баланса) защищены через REPEATABLE READ + FOR UPDATE, поскольку это гарантирует то, что никакая параллельная транзакция не сможет изменить обрабатываемые данные
- Некритичные операции (просмотр каталога) работают на READ COMMITTED, что максимально увеличивает эффективность выполнения этой простой операции
- Четкое разделение зон ответственности, каждая операция имеет свои требования к эффективности и безопасности


**3. Риски deadlock**
- FOR UPDATE может привести к deadlock при неправильном порядке блокировок в случае обработки нескольких таблиц в разном порядке двумя параллельными транзакциями
- Решение: всегда блокировать ресурсы в одном порядке
- Пример: сначала order, потом order_items, потом payment

**4. Простота разработки**
- READ COMMITTED легко понять и отлаживать, поскольку работа ведется с самыми последними изменениями
- REPEATABLE READ + FOR UPDATE требует explicit intent
- Код становится самодокументируемым


**5. Масштабируемость**
- READ COMMITTED хорошо масштабируется
- Локальное использование блокировок не создает bottleneck
- Connection pooling работает эффективно

### Альтернативные подходы

#### Подход 1: Использовать SERIALIZABLE везде

**Плюсы:**
- Максимальная корректность
- Не нужно думать о блокировках
- PostgreSQL автоматически обнаруживает конфликты

**Минусы:**
- Значительное снижение производительности (20-50%)
- Требует retry logic во всех операциях
- Высокий процент rollback при нагрузке
- Сложная отладка serialization failures

**Вывод:** Не рекомендуется для high-load систем.

#### Подход 2: Optimistic Locking (версионирование)

**Реализация:**
```sql
ALTER TABLE orders ADD COLUMN version INTEGER DEFAULT 1;

UPDATE orders 
SET status = 'paid', version = version + 1
WHERE id = '...' AND status = 'created' AND version = 1;

-- Проверить ROW_COUNT, если 0 — конфликт
```

**Плюсы:**
- Нет блокировок на чтение
- Хорошая производительность
- Масштабируется горизонтально

**Минусы:**
- Требует изменения схемы БД
- Требует retry logic в приложении
- Может быть много конфликтов при высокой нагрузке

**Вывод:** Хороший подход для распределенных систем.

#### Подход 3: Advisory Locks

**Реализация:**
```sql
BEGIN;
SELECT pg_advisory_xact_lock(hashtext('order_' || order_id));
-- Критическая секция
COMMIT; -- Блокировка автоматически снимается
```

**Плюсы:**
- Гибкий контроль блокировок
- Можно блокировать логические ресурсы
- Работает на любом уровне изоляции

**Минусы:**
- Легко забыть снять блокировку
- Сложнее отлаживать
- Требует дисциплины от разработчиков

**Вывод:** Полезно для специфичных случаев, но не как основной подход.

### Итоговая рекомендация

Для продакшена маркетплейса используйте **гибридный подход**:

1. **Default isolation level:** `READ COMMITTED`
   - Для обычных операций (чтение каталога, просмотр истории)
   - 95% операций

2. **REPEATABLE READ + FOR UPDATE:** для критичных операций
   - Оплата заказа
   - Изменение баланса
   - Резервирование товара
   - 5% операций

3. **Дополнительно:** Optimistic locking для операций с высокой конкурентностью
   - Обновление счетчиков
   - Изменение рейтингов

**Пример кода:**
```python
async def pay_order(order_id: UUID):
    async with db.transaction(isolation='repeatable_read'):
        # Заблокировать заказ
        order = await db.fetch_one(
            "SELECT * FROM orders WHERE id = $1 FOR UPDATE",
            order_id
        )
        
        if order['status'] != 'created':
            raise OrderAlreadyPaidError()
        
        # Обновить статус
        await db.execute(
            "UPDATE orders SET status = 'paid' WHERE id = $1",
            order_id
        )
        
        # Записать в историю
        await db.execute(
            "INSERT INTO order_status_history (order_id, status) VALUES ($1, 'paid')",
            order_id
        )
```
